Testing Infrastructure as Code
Ensuring Reliability in Terraform and CloudFormation
Cecil Williams
$ terraform test --run-all

The Problem

  • 💰 Infrastructure failures are expensive
  • 📈 Manual testing doesn't scale
  • 🖥️ "It worked on my machine" syndrome
  • 🤔 Traditional software has tests — why not infrastructure?

Testing Pyramid for IaC

Slow, expensive, run less frequently Integration Tests
Medium speed / cost Unit Tests
Fast, cheap, run frequently Static Analysis

What We'll Cover

  • Static analysis — catching errors before deploy (CFN & Terraform)
  • Unit & policy testing — validating logic without real infra
  • Integration testing — deploying real resources, asserting state
  • TDD & BDD — writing tests first, behavior-driven specs
  • When to use each — and a CFN vs. Terraform comparison

Demo 3: cfn-guard Policy as Code

Write your own rules to enforce organisational standards — not just known CVEs

bash
# Install
brew install cloudformation-guard

# s3-rules.guard
let s3_buckets = Resources.*[ Type == "AWS::S3::Bucket" ]

rule S3_PUBLIC_ACCESS_BLOCKED when %s3_buckets !empty {
  %s3_buckets.Properties.PublicAccessBlockConfiguration.BlockPublicAcls == true
    << S3 bucket must block public ACLs >>
  %s3_buckets.Properties.PublicAccessBlockConfiguration.BlockPublicPolicy == true
    << S3 bucket must block public policies >>
}

rule S3_OWNERSHIP_CONTROLS when %s3_buckets !empty {
  %s3_buckets.Properties.OwnershipControls exists
    << S3 bucket must have OwnershipControls configured >>
}

Demo 3: cfn-guard in Action

Validate templates against your rules — bad fails, good passes

bash
$ cfn-guard validate -r s3-rules.guard -d bad-template.yaml
FAILED: S3_PUBLIC_ACCESS_BLOCKED - S3 bucket must block public ACLs
FAILED: S3_OWNERSHIP_CONTROLS   - S3 bucket must have OwnershipControls

$ cfn-guard validate -r s3-rules.guard -d good-template.yaml
PASS: S3_PUBLIC_ACCESS_BLOCKED
PASS: S3_OWNERSHIP_CONTROLS

Terraform — The Testing Landscape

  • More mature testing ecosystem than CloudFormation
  • Multiple layers of testing available
  • Strong community support
  • Built-in testing capabilities

Terraform — Static Analysis

Built-in: terraform validate, terraform fmt, terraform plan  |  Third-party: tflint, tfsec / checkov

hcl
# main.tf
variable "bucket_name" {
  # no type defined!
  default = "my-bucket"
}

resource "aws_s3_bucket" "bad" {
  bucket = var.bucket_name
  # Missing encryption, versioning, etc.
}

Terraform — Policy Testing

Tool: Open Policy Agent (OPA) — brew install opa  |  Write rules in Rego that evaluate against a Terraform plan

rego
# s3_encrypted.rego
deny[msg] {
  resource := input.resource_changes[_]
  resource.type == "aws_s3_bucket"
  not resource.change.after.server_side_encryption_configuration
  msg := "S3 buckets must be encrypted"
}

Terraform — Policy Testing in Action

Export the plan as JSON, then evaluate your Rego rules against it

bash
# 1. Generate plan JSON
$ terraform plan -out=tfplan
$ terraform show -json tfplan > tfplan.json

# 2. Evaluate policy
$ opa eval -d s3_encrypted.rego -i tfplan.json "data.terraform.deny"

# Output (bad config - policy violation):
{
  "result": [{
    "expressions": [{
      "value": ["S3 buckets must be encrypted"]
    }]
  }]
}

# Output (good config - policy passes):
{ "result": [{ "expressions": [{ "value": [] }] }] }

Terratest: Integration Testing

What is Terratest? — Setup: cd test && go mod tidy before running tests

  • Go library for testing infrastructure
  • Deploys real resources to cloud
  • Makes assertions about infrastructure
  • Automatic cleanup with defer
  • Retry logic for eventual consistency
  • Parallel test execution

Terratest: Test Structure

go
func TestS3BucketCreation(t *testing.T) {
  terraformOptions := &terraform.Options{
    TerraformDir: "../",
    EnvVars: map[string]string{
      "AWS_PROFILE": "your-profile",
    },
  }
  defer terraform.Destroy(t, terraformOptions)
  terraform.InitAndApply(t, terraformOptions)
  bucketName := terraform.Output(t, terraformOptions, "bucket_name")
  aws.AssertS3BucketExists(t, "us-east-1", bucketName)
}

Demo 8: Advanced Terratest: Multiple Resources

go
func TestVPCWithSubnets(t *testing.T) {
  // Deploy VPC + Subnets
  terraform.InitAndApply(t, terraformOptions)
  vpcID := terraform.Output(t, terraformOptions, "vpc_id")
  // Verify VPC exists
  vpc := aws.GetVpcById(t, vpcID, "us-east-1")
  assert.Equal(t, "10.0.0.0/16", *vpc.CidrBlock)
  // Verify subnets
  subnets := aws.GetSubnetsForVpc(t, vpcID, "us-east-1")
  assert.Equal(t, 3, len(subnets))
}

BDD for Infrastructure

Behavior-Driven Development meets Infrastructure as Code

What
Gherkin Specs
Describe infrastructure behavior in plain language with Given/When/Then — readable by the whole team
How
godog + Terratest
Cucumber for Go runs the specs; Terratest deploys real AWS resources and makes assertions
Why
Living Documentation
Specs double as docs — if the test passes, the documentation is accurate

Inspired by Stephen Nelson-Smith's "Test-Driven Infrastructure with Chef" (O'Reilly) — which pioneered BDD for infrastructure

Demo 10: BDD S3 Bucket Feature Spec

gherkin
# features/s3_bucket.feature
Feature: Secure S3 bucket
  As a platform engineer
  I need S3 buckets to follow security best practices
  So that our data is protected at rest

  Background:
    Given the Terraform module has been applied

  Scenario: Bucket exists
    Then the S3 bucket should exist in "us-east-1"

  Scenario: Bucket has encryption
    Then the S3 bucket should have "AES256" encryption

  Scenario: Bucket has versioning
    Then the S3 bucket should have versioning enabled

Demo 10: BDD Step Definitions in Go

godog = Cucumber for Go  |  Terratest = AWS assertions

go
func (s *S3TestContext) theTerraformModuleHasBeenApplied(ctx context.Context) error {
  t := GetTestingT(ctx)
  terraform.InitAndApply(t, s.TerraformOptions)
  s.BucketName = terraform.Output(t, s.TerraformOptions, "bucket_name")
  return nil
}

func (s *S3TestContext) theS3BucketShouldHaveVersioningEnabled(ctx context.Context) error {
  t := GetTestingT(ctx)
  v := terraaws.GetS3BucketVersioning(t, "us-east-1", s.BucketName)
  if v != "Enabled" {
    return fmt.Errorf("expected Enabled, got %s", v)
  }
  return nil
}

func InitializeScenario(ctx *godog.ScenarioContext) {
  s := &S3TestContext{}
  ctx.Given(`the Terraform module has been applied`, s.theTerraformModuleHasBeenApplied)
  ctx.Then(`the S3 bucket should have versioning enabled`, s.theS3BucketShouldHaveVersioningEnabled)
}

When to Use Each Test Type

Always
Static Analysis
Run on every commit. Fast feedback. Catches 80% of issues. Zero cost.
Recommended
Policy Testing
Enforce standards. Run in CI/CD. Before production.
Selective
Integration Testing
Critical infrastructure. Before major releases. Complex scenarios.

Real-World Testing Strategy

Pre-commit
  • terraform validate
  • terraform fmt -check
  • tflint
CI/CD Pipeline
  • Security scanning (tfsec/checkov)
  • Policy validation (OPA)
  • Integration tests for critical paths
Before Production
  • Full integration test suite
  • Manual review of plan output

Common Pitfalls

  • ❌ Over-testing — Not every resource needs integration tests
  • ❌ Under-testing — No tests = playing Russian roulette in production
  • ❌ Ignoring costs — Integration tests deploy real resources — set budgets
  • ❌ Not cleaning up — Failed tests can leave orphaned resources

Testing Best Practices

  • ✅ Start with static analysis — Quick wins, easy to implement
  • ✅ Use parallel execution — Speed up integration tests
  • ✅ Test in isolated accounts — Avoid contaminating production
  • ✅ Use meaningful test names — TestS3BucketHasEncryption not Test1
  • ✅ Keep tests focused — One assertion per test when possible
  • ✅ Version your tests — Store with infrastructure code

1
Testing IaC is not optional — Infrastructure failures are expensive
2

3

4

5
TDD works for infrastructure — Write tests first, then code
